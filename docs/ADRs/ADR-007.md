# ADR-007: Calendar Write-Back — Constrained Reminder-to-Calendar Mirroring

**Status:** Accepted
**Date:** 2026-03-28
**Repo:** `woodcreek-agents`

---

## Context

The Woodcreek reminder platform manages household reminders through a durable lifecycle (draft → approve → schedule → deliver → acknowledge). Calendar systems (WorkMail/EWS) provide read-only context for scheduling intelligence. The question is whether approved, scheduled reminders should be **mirrored as calendar events** so household members see them alongside their existing schedule.

## Decision

**Yes — implement constrained, one-way, create-and-delete mirroring for eligible reminders.** Not bidirectional sync.

## Rationale

### Why write-back is useful

- Household members already check their calendar daily. A reminder that exists only in the Woodcreek system is invisible there.
- Mirrored events provide a unified view: "what's on my plate today" includes both external events and Woodcreek reminders.
- Calendar mirroring is **additive UX**, not a replacement for SMS/email/Slack delivery.

### Why NOT full bidirectional sync

- The reminder system is authoritative. Calendar edits must not silently modify reminder state.
- Bidirectional sync requires conflict resolution, which adds fragility.
- One-way mirroring (reminder → calendar) is sufficient for the household use case.

## Eligibility Policy (MVP)

A reminder is eligible for calendar mirroring if **all** conditions are met:

| Condition | Rationale |
|---|---|
| State = `SCHEDULED` or `APPROVED` | Only approved reminders should appear on calendar. |
| Single-target (INDIVIDUAL) | Household-wide reminders would create confusing multi-calendar fanout. |
| Target has active `CalendarIdentity` (WorkMail/EWS) | No identity → no calendar to write to. |
| Intent ≠ `DIGEST` | Digests are email batches, not calendar events. |
| Not already mirrored | Idempotent — don't create duplicates. |
| Governance allows | Kill switch and tier checks apply. |

**Explicitly ineligible:**
- HOUSEHOLD-targeted reminders (multi-calendar fanout deferred)
- DIGEST-intent reminders
- Reminders without a schedule/fire time
- Reminders in DRAFT or terminal states

## Mirror Behavior

### Create
When a reminder becomes SCHEDULED and is eligible:
- Create a calendar event at `schedule.fire_at` with 15-minute duration
- Title: reminder subject
- Body: reminder body + source attribution
- Store the external event ID in a `calendar_mirror` linkage record

### Delete
When a mirrored reminder is CANCELLED:
- Delete the mirrored calendar event
- Update the linkage record to `deleted`

### Update (deferred)
Schedule changes (reschedule) are not supported in this MVP. If a reminder is rescheduled, the old mirror is deleted and a new one created on the next tick. This is simple and avoids partial-update complexity.

### Snooze / Acknowledge
- Snooze: no calendar change (snooze is delivery-level, not schedule-level)
- Acknowledge: no change (event remains as historical record)

## Why WorkMail/EWS First

- Woodcreek already uses WorkMail for email (IMAP ingest, SMTP delivery)
- EWS provides authenticated calendar write with service-level credentials
- No OAuth consent screens or token churn (Tier 1 trust model from ADR-006)
- Gmail/CalDAV/Graph deferred — each adds auth complexity

## Governance Integration

- Write-back calls `governance.evaluate(action_type="calendar_mirror", requires_tier=TIER_1)`
- Kill switch blocks mirroring
- Budget counts mirroring as an autonomous action
- Failures do not block reminder delivery — mirroring is best-effort

## Non-Goals

- No bidirectional sync
- No household-wide multi-calendar fanout
- No calendar-event-originated reminders (read path exists, no write-to-read loop)
- No digest mirroring
- No Gmail/CalDAV write support in this phase

## Implementation Scope (This Phase)

1. `CalendarAdapter.create_event()` and `delete_event()` on the port
2. `WorkMailEwsAdapter` write implementation
3. `calendar_mirror` linkage table
4. `CalendarMirrorService` with eligibility check + create/delete
5. Integration with the control loop (mirror after dispatch)
6. Operator visibility: "was this mirrored?" / "why not?"
7. Tests

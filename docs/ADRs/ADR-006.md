# ADR-006: Household Reminder Orchestrator

**Status:** Proposed
**Date:** 2026-03-28
**Repo:** `woodcreek-agents`

---

## Problem Statement

Woodcreek Agents has the foundations of a home-management agent platform — HOA compliance, maintenance scheduling, warranty tracking, security monitoring, email integration, SMS webhooks, and a RAG pipeline. But reminders, delivery policy, household targeting, acknowledgment, escalation, and calendar-aware orchestration are not yet modeled as a first-class platform capability.

Each agent that needs to notify a household member currently has no shared infrastructure to do so. There is no unified reminder lifecycle, no delivery channel abstraction, no acknowledgment or snooze flow, no escalation policy, and no audit trail. As more agents come online, this gap will produce fragmented notification logic, missed deadlines, and no household-level visibility into "what needs to happen, for whom, by when, and was it done."

This ADR introduces a **Household Reminder Orchestrator** — a shared platform layer that owns the full reminder lifecycle from draft through delivery, acknowledgment, and escalation, with multi-channel delivery, household/individual targeting, and governance boundaries that gate autonomy behind proven audit infrastructure.

---

## Goals

1. **First-class reminder domain.** Reminders are a platform primitive with a durable lifecycle, not per-agent notification logic.
2. **Household-wide and individual targeting.** A reminder can target the household, a specific member, a role, or a subset. Group semantics from day one.
3. **Durable reminder lifecycle.** Draft, schedule, deliver, acknowledge, snooze, escalate, cancel — all state transitions are persisted and auditable.
4. **Multi-channel delivery abstraction.** The delivery layer supports `push`, `sms`, `email`, `in_app`, and Slack, with channel selection driven by preference rules and urgency.
5. **Slack as conversational control surface.** `@woodcreek` interactions in Slack for creating reminders, approving agent drafts, querying schedules, and managing household workflows.
6. **iPhone app / APNs as long-term primary household UX.** Native push notifications are the intended end-state for household reminder delivery and acknowledgment. SMS and email are early delivery and fallback channels.
7. **Provider-neutral calendar abstraction.** A calendar interface from day one, with AWS WorkMail / EWS as the first adapter for Woodcreek-managed identities.
8. **AWS WorkMail as trusted identity baseline.** Woodcreek-managed accounts use service-owned integrations (WorkMail/EWS) that avoid delegated-auth token churn.
9. **Strong auditability and observability.** Every reminder action is logged. Governance infrastructure must be proven before autonomy is granted.
10. **Incremental delivery.** Each phase ships usable value. No big-bang launch.

## Non-Goals

1. **Building the iPhone app immediately.** The architecture anticipates device registration and push delivery, but the app itself is a future deliverable.
2. **Making Gmail OAuth the MVP baseline.** Gmail's delegated-auth lifecycle (consent screens, token refresh, re-consent friction) makes it a poor first adapter. It can be added later as an optional calendar provider.
3. **Unrestricted autonomous writes early.** No agent sends reminders or modifies calendars without human approval until governance infrastructure is proven through sustained use.
4. **Treating reminders as only "calendar notifications."** Reminders originate from calendar events, email-extracted dates, agent observations, household routines, and manual requests. Calendar is one source among many.
5. **Generic "AI calendar assistant" framing.** This is household orchestration — making sure the family doesn't miss what matters — not "schedule a meeting with Bob."
6. **Multi-tenant SaaS.** This serves the Woodcreek household. The domain model shouldn't preclude multi-household, but it's not a design driver.

## Domain Model

### Entity Definitions

| Entity | Description |
|---|---|
| `Household` | Top-level unit. One household per deployment. Contains members, preferences, reminder policies, and connected integrations. |
| `HouseholdMember` | A person in the household. Has contact info (phone, email, Slack ID), delivery preferences, quiet hours, and roles (admin, member, child). |
| `Reminder` | The core unit of work. Subject, body, urgency level, source (agent or user), lifecycle state, and targeting. |
| `ReminderTarget` | Junction between a `Reminder` and one or more `HouseholdMember`s or the household as a whole. Supports individual, role-based, and broadcast targeting. |
| `ReminderSchedule` | When a reminder fires. One-shot (absolute datetime), relative ("3 days before event X"), or recurring (cron expression). |
| `ReminderDelivery` | A record of an attempted delivery to a specific member via a specific channel. Tracks status: queued, sent, delivered, bounced, failed. |
| `ReminderExecution` | The runtime record of a scheduled reminder firing. Links schedule → delivery attempts. Enables retry and escalation. |
| `ReminderAcknowledgement` | A member's response to a delivered reminder. Tracks ack method (Slack reaction, SMS reply, push action, email link), timestamp, and optional note. Includes snooze duration if snoozed. |
| `PreferenceRule` | Per-member or household-wide delivery rules. Examples: "SMS for urgent, email for informational," "no delivery after 10pm unless critical," "digest mode for low-priority." |
| `ReminderContextSource` | An external data source that feeds reminder-relevant context: calendar events, email-derived dates, agent events, manual entries. |
| `CalendarIdentity` | A calendar account linked to a household member or the household. References a `CalendarProvider` and stores connection credentials/tokens. The identity used to read/write calendar data. |
| `CalendarProvider` | A supported calendar backend: AWS WorkMail (EWS), Google Calendar, Apple Calendar (CalDAV), iCal feed, etc. Defines the adapter implementation and auth requirements. |
| `CalendarAccessPolicy` | Per-identity rules governing what the platform can do: read-only vs. read-write, which calendars are visible, sync frequency, and write-back permissions. Defaults to read-only until explicitly upgraded. |
| `MemberDevice` | *(Future — anticipated from Phase 0)* A registered device for a household member. Stores APNs device token, platform (iOS), and notification preferences. Enables push delivery when the iPhone app ships. |

### Entity Relationships

```
Household
  ├─ HouseholdMember[]
  │    ├─ PreferenceRule[]
  │    ├─ CalendarIdentity[] → CalendarProvider
  │    │    └─ CalendarAccessPolicy
  │    └─ MemberDevice[]  (future)
  └─ PreferenceRule[]  (household-wide defaults)

Reminder
  ├─ ReminderTarget[] → HouseholdMember | Household
  ├─ ReminderSchedule
  ├─ ReminderExecution[]
  │    └─ ReminderDelivery[] → HouseholdMember
  │         └─ ReminderAcknowledgement?
  └─ ReminderContextSource?  (originating source)

CalendarProvider
  └─ CalendarIdentity[]  (one provider, many identities)
```

## Reminder Lifecycle States

```
  ┌──────────┐
  │  draft    │  Agent or user creates a reminder
  └────┬─────┘
       │ (schedule attached)
  ┌────▼──────────────┐
  │  scheduled         │  Waiting for fire time
  └────┬──────────────┘
       │ (fire time reached or immediate)
  ┌────▼──────────────┐
  │  pending_delivery  │  Delivery attempts in progress
  └────┬──────────────┘
       │ (at least one channel confirmed)
  ┌────▼──────────────┐
  │  delivered         │  Reached the target member(s)
  └────┬──────────────┘
       │
       ├─── acknowledged   Member confirmed receipt (ack via Slack/SMS/push/email)
       ├─── snoozed        Member deferred ("snooze 30 min") — returns to pending_delivery after interval
       └─── escalated      No ack within window — re-deliver via alternate channel or member

  Terminal states:
    cancelled  — manually cancelled by user or agent
    failed     — all delivery attempts and escalation paths exhausted
```

**Agent-authored reminders** may include a `pending_approval` gate between `draft` and `scheduled` (when approval flow is enabled). The household member approves via Slack or another surface before the reminder enters `scheduled`.

**User-created reminders** skip approval and go directly from `draft` to `scheduled`.

**Recurring reminders** generate a new `ReminderExecution` per occurrence; the parent `Reminder` stays in `scheduled` state.

**Snoozed reminders** re-enter `pending_delivery` after the snooze interval expires, creating a new `ReminderExecution`.

## Delivery Channels

The delivery abstraction is designed from the start to support all channels, even if some are implemented in later phases:

| Channel | Type | Provider | Phase | Notes |
|---|---|---|---|---|
| `sms` | Delivery | Telnyx | Phase 3 | Urgent reminders, fallback, escalation, external reach. Ack via reply. |
| `email` | Delivery | woodcreek.me SMTP | Phase 3 | Rich context, digests, archival. Ack via link or reply. |
| `slack` | Control + Delivery | Slack API | Phase 2-3 | Primary control surface. Also delivers operator/admin notifications with interactive buttons. |
| `push` | Delivery | APNs (iPhone) | Future | Long-term primary household notification and ack surface. |
| `in_app` | Delivery | Web dashboard | Future | Reminder inbox, household activity view. |

### Channel Roles

Each channel has a distinct role in the household orchestration model:

**Slack — Conversational control surface**
The primary way users interact with the reminder system. Create reminders, approve agent drafts, query household schedules, manage preferences. Slack is where `@woodcreek` lives. It is also used for operator/admin notifications (approval requests, escalation alerts, governance summaries). Slack is a command/control surface first and a notification channel second.

**Telnyx SMS — Urgent delivery, fallback, escalation**
SMS has external reach (works without internet/app) and high open rates. It is the channel for urgent reminders, for reaching members who haven't responded on other channels, and for escalation when ack deadlines pass. SMS is not the primary conversational surface — Slack is.

**Email (woodcreek.me) — Rich context, digests, archival**
Email carries detailed reminder content, weekly household digests, and serves as the archival record. It is the right channel for informational reminders that don't need immediate action. Email is also the channel for context that's too long for SMS or push.

**iPhone app / APNs push — Long-term primary household notification**
Everyone in the household is on iPhone. Native push notifications are the intended end-state for time-sensitive reminder delivery and quick acknowledgment (ack/snooze via notification actions). Until the app ships, SMS and Slack fill this role.

**In-app (web dashboard) — Household activity view**
Future reminder inbox and activity feed. Not a delivery channel per se — it's where members go to see the full picture.

### Channel Selection Logic

1. Check `PreferenceRule` for the target member and urgency level.
2. If no rule matches, use household defaults: Slack for routine, SMS for urgent, email for informational/digest.
3. During quiet hours, hold delivery unless urgency is critical (then SMS).
4. On delivery failure, fall back to next preferred channel.
5. When push (APNs) is available, it becomes the default for time-sensitive delivery; SMS becomes pure fallback/escalation.

## Slack Interaction Model

Slack is the first-class conversational control surface for the Woodcreek household. Members interact with agents via `@woodcreek` in a household Slack workspace.

### Example Interactions

| Command | What happens |
|---|---|
| `@woodcreek what does tomorrow look like?` | Returns a household schedule summary: calendar events, pending reminders, and known commitments for all members. |
| `@woodcreek remind the family about soccer at 5` | Creates a reminder targeting the household, scheduled for the next occurrence of 5pm, delivered via each member's preferred channel. |
| `@woodcreek who has conflicts Thursday night?` | Queries calendar context (via WorkMail adapter) and returns per-member availability for Thursday evening. |
| `@woodcreek show reminders waiting for approval` | Lists agent-drafted reminders in `pending_approval` state with approve/reject buttons. |
| `@woodcreek snooze the trash reminder for 2 hours` | Transitions a delivered reminder to `snoozed` state, re-delivers after 2 hours. |
| `@woodcreek show what was sent this week` | Returns a delivery audit summary: reminders sent, channels used, ack status. |

### Design Principles

- Slack is the **command/control surface**, not the only notification channel. Members don't need to live in Slack to receive reminders — SMS, email, and eventually push handle delivery.
- Agent-authored reminders surface in Slack for approval. The approval UX uses Slack interactive messages (buttons, not free-text parsing).
- Slack threads are used for reminder context (e.g., the original event that triggered a reminder, delivery status updates).
- The Slack bot is the same `@woodcreek` agent identity used across the platform — not a separate reminder-only bot.

## iPhone App End-State

### Product Direction

Everyone in the Woodcreek household is on iPhone. The long-term primary notification and acknowledgment surface is a native iOS app with APNs push notifications.

### What This Means for Architecture

- **`MemberDevice` entity** is anticipated from Phase 0. The schema includes device token, platform, and member association, even though push delivery is implemented later.
- **Push as a delivery channel** is a first-class entry in the delivery abstraction from the start. The `push` channel type exists in the enum and preference rules even before the APNs adapter ships.
- **Notification actions** (ack, snooze, view details) map directly to reminder lifecycle transitions. The push UX mirrors what Slack interactive messages provide today.
- **Household activity feed** in the app replaces the `in_app` delivery channel concept — it's the canonical view of reminder history and status.

### Transition Plan

1. **Now:** Slack for control, SMS for urgent delivery, email for context/archive.
2. **When the app ships:** Push becomes the default time-sensitive delivery channel. SMS becomes pure fallback/escalation. Slack remains the admin/control surface.
3. **Steady state:** Push + Slack + email cover 95% of use cases. SMS is the "last resort" channel for critical escalation.

## Calendar Strategy

### Provider-Neutral Interface

A `CalendarAdapter` interface is defined from Phase 0. All calendar operations go through this abstraction, regardless of backend provider.

```
CalendarAdapter
  ├─ list_events(identity: CalendarIdentity, range: DateRange) → Event[]
  ├─ get_event(identity: CalendarIdentity, id: str) → Event
  ├─ create_event(identity: CalendarIdentity, event: Event) → Event   # Phase 4+
  └─ delete_event(identity: CalendarIdentity, id: str) → void         # Phase 4+

Event
  ├─ id, title, description
  ├─ start, end, all_day
  ├─ recurrence_rule?
  ├─ source: CalendarProvider
  ├─ owner: CalendarIdentity
  └─ metadata: dict
```

### AWS WorkMail / EWS as First Adapter

| Decision | Rationale |
|---|---|
| **WorkMail/EWS is the MVP calendar adapter.** | Woodcreek already uses AWS infrastructure. WorkMail provides calendar + email in a single managed service. EWS API supports full read/write calendar access with service-level auth. |
| **WorkMail-backed identities are the trusted default** for Woodcreek-managed household accounts. | Service-owned integrations have no delegated-auth token churn. No consent screens, no refresh token expiry, no re-authorization friction. The platform controls the identity lifecycle. |
| **Gmail OAuth is explicitly deferred.** | Gmail's OAuth consent flow, token refresh lifecycle, and re-consent requirements create ongoing operational friction. It is a poor MVP baseline for a system that must reliably read calendar data. Gmail can be added as an optional adapter later for household members who use it as their personal calendar. |
| **Do not let alias-based mail routing define the calendar identity model.** | `CalendarIdentity` is a separate concept from email aliases. A member may have a woodcreek.me email address for delivery but use a WorkMail calendar identity (or eventually a Google identity) for calendar access. |

### Planned Adapters

| Adapter | Read | Write | Phase | Priority |
|---|---|---|---|---|
| AWS WorkMail (EWS) | Phase 1 | Phase 4 | MVP | High — first adapter, Woodcreek-managed identity |
| iCal feed (URL) | Phase 1 | — | MVP | High — read-only, works with any published calendar |
| Email-derived dates | Phase 1 | — | MVP | High — leverages existing email RAG pipeline |
| Manual entries | Phase 0 | Phase 0 | Immediate | — |
| Google Calendar (OAuth) | Future | Future | Deferred | Optional — for members with personal Google calendars |
| Apple Calendar (CalDAV) | Future | Future | Deferred | Optional |

Calendar events flow into `ReminderContextSource` and inform reminder suggestions. They do not replace the reminder domain model.

## Identity and Integration Trust Model

### Principle

Core reminder reliability must not depend on delegated-auth token churn from external providers. Woodcreek-managed identities should prefer service-owned integrations with lower re-consent friction.

### Trust Tiers

| Tier | Integration type | Example | Auth model |
|---|---|---|---|
| **Tier 1 — Service-owned** | Platform controls the identity and credentials. | AWS WorkMail (EWS service account) | Service credentials, no user consent flow. Highest reliability. |
| **Tier 2 — Delegated with durable tokens** | User authorizes once, platform maintains long-lived access. | Google Calendar (OAuth with offline refresh) | Refresh token with periodic re-consent risk. Acceptable for optional adapters. |
| **Tier 3 — Delegated with session tokens** | User must re-authorize frequently. | Services with short-lived tokens or no offline access. | Not suitable for core reminder delivery. |

### Implications

- The **reminder delivery pipeline** (SMS via Telnyx, email via woodcreek.me SMTP) uses Tier 1 integrations — the platform owns the accounts and credentials.
- The **calendar read pipeline** (WorkMail/EWS) uses Tier 1 for the primary adapter.
- **External calendar providers** (Google, Apple) are Tier 2 — optional, additive, and non-blocking for core functionality.
- If a Tier 2 integration's token expires, the system degrades gracefully (missing calendar context, not missing reminder delivery).

## Escalation and Acknowledgement Model

### Acknowledgement

- Each `ReminderDelivery` can have an associated `ReminderAcknowledgement`.
- Ack methods: Slack reaction/button, SMS reply ("OK"), push notification action, email link click, dashboard button.
- Ack is optional for informational reminders, expected for action-required reminders.

### Snooze

- Members can snooze a delivered reminder for a specified duration ("snooze 30 min", "snooze until tomorrow").
- Snooze transitions the reminder to `snoozed` state and creates a new `ReminderExecution` for re-delivery after the interval.
- Snooze is available via Slack command, SMS reply, and (future) push notification action.

### Escalation Rules

1. **Timeout:** If no ack within a configurable window (default: 2 hours for normal, 30 minutes for urgent), escalate.
2. **Channel escalation:** Re-deliver via alternate channel (e.g., SMS after Slack got no response).
3. **Member escalation:** If target member doesn't ack, notify household admin or alternate member.
4. **Max escalation depth:** Configurable (default: 2 escalation attempts before marking `failed`).
5. **Escalation is logged:** Every escalation step is a new `ReminderDelivery` record linked to the same `ReminderExecution`.

## Integration Touchpoints

### Email (woodcreek.me)

- **Inbound:** Email RAG pipeline extracts dates, deadlines, and action items → creates `ReminderContextSource` entries → agents propose reminder drafts.
- **Outbound:** `ReminderDelivery` sends reminder emails via SMTP. Emails include ack link and reminder metadata. Best for digests and detailed context.
- **Threading:** Reminder emails use consistent `Message-ID` / `In-Reply-To` headers for conversation threading.

### SMS (Telnyx)

- **Inbound:** SMS webhook receives ack/snooze replies and natural-language reminder requests → routes to reminder service.
- **Outbound:** `ReminderDelivery` sends SMS via Telnyx API. Messages are concise with ack instructions ("Reply OK to acknowledge, SNOOZE to defer 1hr").
- **Role:** Urgent delivery, fallback when other channels fail, escalation, and external reach (works without internet/app).

### Slack

- **Inbound:** `@woodcreek` bot receives commands, queries, and approval actions via Slack Events API and interactive messages.
- **Outbound:** Reminder approval requests, delivery confirmations, schedule summaries, and governance alerts posted to household channels or DMs.
- **Role:** Primary conversational control surface. Not the only notification channel — members don't need to live in Slack.

### AWS WorkMail / EWS

- **Calendar read:** `CalendarAdapter` implementation reads events via EWS API using service-level credentials.
- **Calendar write (Phase 4):** Approved reminders optionally create calendar events with source attribution.
- **Identity:** WorkMail-backed `CalendarIdentity` entries are the trusted default for Woodcreek-managed accounts.

### Inter-Agent Communication

- Agents emit structured events (e.g., `warranty.expiring`, `hoa.deadline.approaching`, `camera.offline`) via the event bus.
- The reminder orchestrator subscribes to these events and applies trigger rules to create reminder drafts.
- Agents can also call the reminder API directly to create drafts for specific members.

### LangGraph

- The reminder lifecycle (draft → schedule → deliver → ack/escalate) maps naturally to a durable LangGraph graph.
- PostgreSQL checkpointing (already in place) persists reminder workflow state across restarts.
- Long-running reminders (e.g., recurring weekly) are durable workflows, not ephemeral cron jobs.

## Observability, Auditability, and Governance

### Audit Trail Requirements

Every state transition in the reminder lifecycle produces an audit record:

| Field | Description |
|---|---|
| `timestamp` | When the action occurred |
| `actor` | Who or what performed the action (user ID, agent name, system) |
| `action` | What happened (created, approved, scheduled, dispatched, delivered, acked, snoozed, escalated, cancelled, failed) |
| `reminder_id` | The reminder this action belongs to |
| `target_member_id` | The household member affected (if applicable) |
| `channel` | Delivery channel used (if applicable) |
| `metadata` | Additional context (failure reason, escalation step, snooze duration, etc.) |

### Observability

- **Metrics:** Reminder count by state, delivery success/failure rate by channel, mean time to ack, escalation rate, snooze frequency.
- **Alerting:** Operator alert (via Slack) if delivery failure rate exceeds threshold, if escalation queue grows, or if a member has N+ unacked reminders.
- **Structured logging:** All reminder operations emit structured log events.
- **Dashboard:** Household admin view of reminder activity, delivery status, and governance metrics (Phase 3+).

### Retention

- Audit trail records are retained indefinitely (append-only).
- Reminder content (subject, body) follows the data retention policy from ADR-002 (when defined).
- Delivery metadata (phone numbers, email addresses, device tokens) is treated as PII per household data policy.

### Governance and Autonomy Boundaries

#### Core Principle

**No agent sends a reminder to a household member without explicit human approval until the governance infrastructure has been proven through sustained use.**

#### Autonomy Tiers

| Tier | Capability | Prerequisite |
|---|---|---|
| **Tier 0** (Phase 0-2) | Agents draft reminders. Humans approve via Slack and send. | Domain model and approval flow in place. |
| **Tier 1** (Phase 3) | Agents draft, delivery system sends on approval. Escalation follows rules. | Multi-channel delivery, ack model, audit trail proven in production. |
| **Tier 2** (Phase 5) | Agents auto-send pre-approved reminder types without per-instance approval (e.g., recurring maintenance). | Governance dashboard, rate limiting, anomaly detection, household admin review of production data. |
| **Tier 3** (Phase 5+) | Agents escalate and re-route autonomously based on ack status. | Full audit review, opt-out mechanisms, periodic governance review. |

#### Guardrails (All Tiers)

- **Rate limit:** Max reminders per member per day (configurable, default: 10).
- **Quiet hours:** No delivery during configured quiet hours unless urgency is critical.
- **Opt-out:** Any member can mute, snooze, or opt out of non-critical reminders.
- **Kill switch:** Household admin can disable all autonomous actions instantly via Slack command or dashboard.
- **Anomaly detection:** Flag unusual patterns (sudden spike in reminders, repeated escalations to same member).
- **Approval checkpoints:** Calendar write-back and any destructive action (cancel all, bulk modify) require explicit approval even at higher autonomy tiers.

#### Governance Review

- Monthly automated summary of all autonomous actions delivered to household admin via Slack and email.
- Quarterly review of autonomy tier assignments (can tighten or loosen per agent).

## Implementation Sequencing Recommendation

The following order delivers the fastest path to user value while building trust for later autonomy:

### 1. Domain Model and Reminder Lifecycle (Phase 0)
Define and persist `Household`, `HouseholdMember`, `Reminder`, `ReminderTarget`, `ReminderSchedule`, `CalendarIdentity`, `CalendarProvider`, and related entities. Include `MemberDevice` as a schema placeholder for future push. Ship the PostgreSQL schema, Pydantic/TS models, shared library, and provider-neutral `CalendarAdapter` interface.

**Delivers:** Foundation for everything. No runtime behavior yet.

### 2. Reminder Draft/Create/Schedule Flows (Phase 2)
Build the reminder creation API and basic scheduling engine. Users create reminders via CLI, SMS, or Slack. Agents create drafts that surface for approval in Slack. Household and individual targeting works from the start.

**Delivers:** First user-visible capability. "Remind me to check the mailbox filter on Saturday" works.

### 3. Outbound Delivery via SMS and Email (Phase 3, partial)
Wire Telnyx SMS and woodcreek.me email delivery. Start with simple one-shot delivery, add ack tracking, then escalation. This is the highest-value milestone — the household starts receiving real reminders on real channels.

**Delivers:** Reminders become actionable. Household members get SMS and email.

### 4. Slack Command/Control Workflows (Phase 2-3)
Implement `@woodcreek` Slack bot: reminder creation, approval buttons, schedule queries, snooze commands. Slack becomes the daily control surface for the household.

**Delivers:** Conversational UX. "What does tomorrow look like?" works.

### 5. Read-Only Calendar Context with WorkMail/EWS (Phase 1)
Implement the WorkMail/EWS calendar adapter and iCal feed reader. Agents see upcoming events and suggest relevant reminders. Email-derived dates from existing RAG serve as an interim context source until calendar adapters are wired.

**Delivers:** Calendar-aware reminder suggestions. Agents know about upcoming events.

### 6. Event-Driven Triggers (Phase 4)
Connect the agent event bus. Warranty expirations, compliance deadlines, and camera alerts automatically generate reminder drafts. Calendar write-back syncs approved reminders to WorkMail.

**Delivers:** Proactive reminders. The platform anticipates needs instead of waiting for manual input.

### 7. iPhone/Push-Oriented Delivery Evolution (Future)
Implement `MemberDevice` registration, APNs adapter, and push notification actions (ack, snooze). Push becomes the default time-sensitive channel; SMS becomes pure fallback.

**Delivers:** Native household notification UX. The "pull out your phone and tap" experience.

### 8. Limited Autonomy After Guardrails (Phase 5)
After months of production audit data and household admin review, begin allowing pre-approved reminder types to send without per-instance human approval. Expand autonomy tiers gradually.

**Delivers:** Hands-off operation for trusted, recurring reminder patterns.

**Why this order:** Steps 1-3 get real reminders to real people as fast as possible. Slack (step 4) adds the daily UX layer. Calendar context (step 5) enriches but doesn't block — a manually created reminder that sends via SMS is more useful than a calendar-aware system that can't send anything. Event-driven triggers (step 6) make the system proactive. Push (step 7) upgrades the delivery UX. Autonomy (step 8) comes last because it requires trust earned from steps 1-7.

## Open Questions for Future ADRs

- **ADR-002 (Data & Privacy):** PII handling for phone numbers, email addresses, device tokens, and reminder content. Retention policies for audit trail vs. content.
- **ADR-003 (Integrations):** Detailed design for calendar adapter interface, email threading protocol, event bus schema, and Slack app manifest.
- **Reminder template system:** Should reminders support templates with variable fields (e.g., "HOA compliance deadline for {item}")? Likely yes, scope TBD.
- **Natural language reminder creation:** "Remind me to X on Y" parsing from Slack and SMS. Intent extraction design.
- **Conflict detection:** What happens when two agents propose conflicting reminders for the same event? Resolution/dedup strategy TBD.
- **WorkMail provisioning:** Detailed setup for WorkMail organization, mailboxes, and EWS service credentials. May warrant its own integration guide rather than an ADR.
- **Slack app architecture:** Bot token scopes, event subscriptions, interactive message payloads, channel conventions. Likely an ADR or integration spec.
- **Push notification design:** APNs certificate management, notification payload schema, background refresh strategy. Deferred until iPhone app work begins.

## Alternatives Considered

1. **Per-agent notification logic:** Each agent handles its own reminders. Rejected: inconsistent delivery, no shared ack/escalation, duplicate effort, no household-level view.
2. **Calendar-centric model (Google Calendar as system of record):** Rejected: couples to a single provider with token-churn friction, misses SMS/Slack as primary surfaces, doesn't model household semantics.
3. **Gmail OAuth as MVP calendar baseline:** Rejected: consent screen UX, token refresh lifecycle, and re-consent requirements create operational friction. WorkMail/EWS with service-owned credentials is more reliable for an MVP.
4. **Third-party reminder service (Todoist, Apple Reminders, etc.):** Rejected: no control over delivery channels, can't integrate with agent events, no household-level governance.
5. **SMS-only delivery:** Rejected: SMS is expensive at scale, lacks rich formatting, and doesn't support interactive approval UX. SMS is the right fallback/escalation channel, not the primary surface.
